1. v1.0 Data Architecture — Static Site
This is a static single-page application. There is no database, no server, and no API calls at runtime. All data is defined as JavaScript module exports in /src/data/ and imported directly into React components at build time.
This is not a limitation — it is a deliberate architectural choice. Static data means: zero latency data access, no CORS issues, no API rate limits, and trivial deployment. The data schema below defines the shape of these static JavaScript objects.

3. v2.0 Backend Architecture Roadmap
If the business grows to require dynamic content management, the following backend architecture is recommended. This is out of scope for v1.0 but designed to be non-breaking — the React frontend would migrate from static imports to API calls with minimal refactoring.
3.1 Recommended Stack
Backend: Node.js + Express OR Next.js API routes
Database: PostgreSQL (relational — products have categories, orders have line items)
ORM: Prisma (type-safe, excellent migration tooling)
Auth: NextAuth.js or Clerk (admin CMS access)
Storage: Cloudinary (image CDN with WebP auto-conversion)
Hosting: Vercel (frontend) + Railway or Supabase (backend/DB)
3.2 v2.0 Database Schema (PostgreSQL / Prisma)
model MenuItem {
  id          String    @id @default(cuid())
  name        String
  price       Float
  description String
  imageUrl    String    // Cloudinary CDN URL
  category    Category
  isFeatured  Boolean   @default(false)
  isHeroCake  Boolean   @default(false)
  layoutId    String?
  slotIndex   Int?
  createdAt   DateTime  @default(now())
}
model Testimonial {
  id           String   @id @default(cuid())
  reviewerName String
  quote        String
  stars        Int
  isPublished  Boolean  @default(false)
  createdAt    DateTime @default(now())
}
enum Category { SIGNATURE SEASONAL CUSTOM WEDDING }
